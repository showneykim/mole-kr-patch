# 04. 번역 파이프라인

> Game.locres의 1,300개 문자열(일본어 기준 33,687자, ≈67페이지)을, 멀티에이전트 워크플로로 **번역 → 3렌즈 병렬 교차검증 → 교정**해 한국어로 완역한 과정. 결과는 [`data/translations.json`](../data/translations.json)(빌드용)과 [`data/translations.tsv`](../data/translations.tsv)(검토용)에 고정되어 있다.

관련 문서: 게임 구조와 locres 포맷은 [02-reverse-engineering.md](02-reverse-engineering.md), 폰트 교체는 [03-fonts.md](03-fonts.md), 엔진 키 이름·원작 버그는 [05-engine-keys-and-bugs.md](05-engine-keys-and-bugs.md), 의사결정 요약은 [DECISIONS.md](DECISIONS.md) 참조.

---

## 1. 분량과 성격

`Mole/Content/Localization/Game/ja/Game.locres`에는 게임 본문 텍스트 **1,300개**가 들어 있다.

| 지표 | 값 |
|---|---|
| 문자열 수 | 1,300개 (모두 고유 — 중복 0) |
| 일본어 글자 수 합계 | 33,687자 (≈67페이지) |
| 짧은 UI(≤15자) | 약 747개 |
| 긴 서사(>80자) | 68개 |

즉, 메뉴·HUD·조작 라벨 같은 짧은 항목이 다수이고, 일기·편지·신문기사·터미널 로그 같은 긴 서사가 일부 섞인 구성이다. M.O.L.E은 심리 호러 서사가 핵심인 게임이라, 짧은 분량이라도 톤과 용어 일관성이 품질을 좌우한다.

## 2. 원문 확보 — 영어까지 함께

우리가 채우는 슬롯은 일본어(ja)지만, 게임의 **소스 언어는 영어**다([06-verification.md](06-verification.md) 참고). 번역 정확도를 높이기 위해 **일본어 + 영어 원문을 함께** 봤다.

locres는 언어별로 독립된 문자열 테이블을 가지며 인덱스가 서로 다르다(ja 1,300 / en 1,795, 정렬도 불일치). 그러나 각 엔트리의 **키 GUID는 언어 간 동일**하므로, ja와 en의 키 GUID로 조인해 같은 항목의 (일본어, 영어) 쌍을 만들었다.

```
ja: keyGUID → ja_index → 일본어 텍스트
en: keyGUID → en_index → 영어 텍스트
join on keyGUID  ⇒  (영어, 일본어)  →  한국어 번역
```

번역 원칙: **ja 의미를 우선**(실제 게임에 들어갈 슬롯)하되, **en을 보조 참고**해 중역으로 인한 의미 왜곡을 줄인다. 둘이 충돌하면 ja를 기준으로 판정한다.

> 예: index 0 `戻る`는 영어가 `Backward`(메뉴 "뒤로"가 아니라 이동 "후진"). 영어 원문 덕분에 맥락을 잡아 "후진"으로 교정했다.

## 3. 맥락(register)별 분류와 배치

알파벳 정렬로 흩어진 문자열을 **문체(register) 8종**으로 분류하고, 각 분류에 맞는 어투 지침을 부여했다. 그런 다음 약 45개씩 묶어 **33배치**로 나눴다.

| 카테고리 | 문체 지침 요약 |
|---|---|
| `UI` | 짧은 메뉴/버튼/HUD/조작 라벨. 명사형 또는 해라체. 게이지·단위 기호 보존 |
| `NUMERIC` | 수치·코드·날짜·좌표. 숫자/단위/16진코드는 그대로, 포함된 단어만 한국어로 |
| `SYSTEM` | 터미널/시스템 메시지. 건조한 기계체. `>>` `[//]` `[ENTER]` 등 기호 보존 |
| `ITEM` | 소지품/오브젝트 설명. 간결한 묘사체(한다체·명사종결), 불길한 톤 |
| `DIALOGUE` | 인물 대사·독백. 자연스러운 구어체, 화자 감정 반영, 인용부호 보존 |
| `DIARY` | 일기·편지·개인 기록. 1인칭 회고체, 날짜 형식·줄바꿈 유지 |
| `DOC` | 신문기사·공식 문서·보고·공지. 문어체/보도체 |
| `MISC` | 일반 내레이션·안내. 문맥에 맞춰 자연스럽게, 호러 분위기 유지 |

이렇게 묶으면 한 에이전트가 같은 문체의 항목을 일관되게 처리하고, 검수자도 같은 잣대로 본다.

## 4. 멀티에이전트 워크플로

번역 본체는 [`tools/wf_translate.reference.js`](../tools/wf_translate.reference.js)에 담긴 워크플로다(참고용 스크립트). 33배치를 **파이프라인**으로 흘리며, 각 배치는 다음 3단계를 독립적으로 통과한다.

```
배치 ──▶ ① 번역 ──▶ ② 3렌즈 병렬 교차검증 ──▶ ③ 교정/확정
        (1 에이전트)   (3 에이전트 동시)        (1 에이전트)
```

### ① 번역
글로서리와 배치 원문(일본어 + 영어)을 읽고, 문체 지침에 맞춰 번역한다. 모든 항목을 인덱스 단위로 빠짐없이 출력하도록 강제(스키마)한다.

### ② 3렌즈 병렬 교차검증 (검수 게이트)
같은 번역 결과를 **서로 다른 관점의 검수자 3명**이 동시에 본다. 각자 문제 항목만 지적(없으면 빈 배열)한다.

| 렌즈 | 중점 |
|---|---|
| 정확성 | ja 우선·en 참고로 의미 대조. 오역/누락/추가/왜곡 |
| 용어·문체 | 글로서리 고유명사·용어 일관성, 카테고리 register 준수 |
| 포맷·자연스러움 | `{/n}`·줄바꿈·`[ENTER]`·코드·숫자 보존, 맞춤법·어색함 |

### ③ 교정/확정
검수 지적을 반영해 해당 항목을 수정하고(지적 없는 항목은 유지), 보존 규칙을 재점검한 뒤 배치 최종본을 확정한다.

### 규모
- 33배치 × (번역 1 + 검수 3 + 교정 1) = **약 165 에이전트**
- **검수 지적 105건**을 반영
- **1,300개 전량** 번역 완료

## 5. 글로서리 — 공식 용어 채택

용어 일관성의 기준은 [`tools/glossary.md`](../tools/glossary.md)다. **공식 스팀 한국어 상점 페이지**(한국어 지원은 없지만 설명문은 한국어로 존재)의 용어를 그대로 채택했다.

| 원어(JA / EN) | 한국어 |
|---|---|
| モール / Mole(기계) | 두더지 |
| 掘削艇 / driller | 굴착기 |
| 地航行士 / Navigator | 항해사 |
| 乗員 / crew | 승조원 |
| ストラータ / Strata | 스트라타 |
| 神経カセット / neural cassette | 신경 카세트 |

인물명(스테판 아다멘코, 페트로 카민스키, 알렉세이 파트로프, 흐리첸코, 빅토르, 요세프 등)도 표기를 고정했다. 문체는 호러 분위기를 살리되, 공식 슬로건은 해라체로 통일했다: **정신을 놓지 마라 / 굴착기를 운용하라 / 살아남아라 / 진실을 밝혀라.**

## 6. 보존 규칙 (절대 변형 금지)

게임 엔진/위젯이 해석하는 토큰과 데이터는 번역 중 그대로 유지한다.

- `{/n}` — 게임의 줄바꿈 토큰
- 실제 줄바꿈 문자(위치 유지)
- `[ENTER]`, `[001-032]` 같은 라벨, 버전(`[버전 79.0411]`)·주파수(`[12Hz-70Hz]`)·좌표
- 단위(m, °C, Hz), 16진 에러코드(`0x000404`)

## 7. 프로그램 후검증

에이전트 검수에만 의존하지 않고, **기계적으로 다시 검증**했다(태그 보존은 사람보다 코드가 정확하다).

- `{/n}` 개수 일치, `[ENTER]`·`[001-032]`·16진 에러코드 보존
- 실제 줄바꿈 수 큰 차이 탐지, 빈 번역·누락 인덱스 탐지
- **고유명사 일관성**: 일본어 원문에 고유명사가 있는 항목의 한국어가 표준 표기를 쓰는지 대조

최종 결과: **누락 0 / 검증 이슈 0 / 고유명사 불일치 0.** 핵심 용어 빈도도 정상(두더지 8·굴착기 25·항해사 26·카세트 59회 등)으로 확인했다.

## 8. 결과 데이터와 재현

번역 결과는 두 형태로 고정된다.

| 파일 | 용도 | 형식 |
|---|---|---|
| [`data/translations.json`](../data/translations.json) | **빌드용** | `{ "일본어 원문": "한국어" }` (1,300) |
| [`data/translations.tsv`](../data/translations.tsv) | **검토·수정용** | `index · key_guid · english · japanese · korean` |

- **키잉 방식**: 빌드는 인덱스가 아니라 **일본어 원문 문자열**을 키로 매칭한다(`ja → ko`). 인덱스는 게임 업데이트로 밀릴 수 있지만, 원문 문자열 매칭은 그에 견고하다. 1,300개 일본어 원문이 모두 고유해 1:1 매핑이 보장된다(충돌 0건 확인).
- **워크플로는 참고용**이다. 번역 결과 자체는 `data/`에 동결되어 있으므로, 빌드([`tools/build.py`](../tools/build.py))는 워크플로를 다시 돌리지 않고 `data/translations.json`을 그대로 적용한다.
- 오역·어색한 표현 발견 시 `data/translations.tsv`(또는 json)를 고치고 다시 빌드하면 즉시 반영된다. 커뮤니티 교정에 열려 있다.
