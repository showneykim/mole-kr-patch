# 05. 엔진 키 표시명과 원작 버그

> 게임 내 키 안내(예: "달리기 / 左 Shift")와 복호화 프롬프트의 "スペース"는 우리가 번역한 `Game.locres`가 아니라 **언리얼 엔진의 키 표시명**에서 온다. 또한 원작 일본어 로컬라이즈에는 **리터럴 `\r`** 이 박힌 버그가 13곳 있었다. 이 문서는 그 추적과 해결을 기록한다.

관련 문서: [02. 리버스 엔지니어링](02-reverse-engineering.md) · [04. 번역 파이프라인](04-translation.md) · [06. 전수 검증](06-verification.md)

---

## 1. 증상: 한자/일본어로 남는 키 이름

전체 번역(1300개)을 적용한 뒤에도 두 곳에서 일본어가 남았다.

| 화면 | 보이던 것 | 정체 |
|---|---|---|
| 튜토리얼 | `달리기` / **`左 Shift`** | "Left Shift" 키 이름 |
| 복호화 단말 | `데이터 복호화는 다음을 눌러라:` **`スペース`** | "Space" 키 이름 |

`左 Shift`, `スペース`는 `Game.locres` 어디에도 없었다. 즉 게임 텍스트가 아니라 **엔진이 제공하는 `FKey` 표시명**이다.

이 표시명은 다음 파일에 들어 있다.

```
Engine/Content/Localization/Engine/ja/Engine.locres   (34,552 문자열)
```

이 파일은 **대부분 에디터 UI 문자열**이며, 런타임에 실제로 보이는 것은 극히 일부(키 표시명)뿐이다. 따라서 전체를 번역하지 않고 **실제로 표시되는 키 이름만** 골라 번역한다. (이 파일도 레거시 `Mole-Windows.pak` 안에 있어 `_P` 패치로 덮어쓸 수 있다 — [02 문서](02-reverse-engineering.md) 참조.)

## 2. 1차 번역: InputKeys "긴 이름" 블록

키 표시명은 `InputKeys` 네임스페이스에 모여 있고, 인덱스 기준 대략 **11461–11669** 구간에 연속 배치돼 있다. 여기서 일본어로 된 키 이름 **109개**를 한국어로 옮겼다.

| 키 | 인덱스 | 일본어 | 한국어 |
|---|---|---|---|
| `LeftShift` | 11552 | 左 Shift | 왼쪽 Shift |
| `RightShift` | 11652 | 右 Shift | 오른쪽 Shift |
| `LeftControl` | 11549 | 左 Ctrl | 왼쪽 Ctrl |
| `SpaceBar` | 11656 | スペースバー | 스페이스바 |
| `LeftMouseButton` | 11550 | マウスの左ボタン | 마우스 왼쪽 버튼 |
| `MouseWheelAxis` | 11598 | マウスホイール軸 | 마우스 휠 축 |
| `Gamepad_FaceButton_Bottom` | 11491 | ゲームパッドAボタン | 게임패드 A 버튼 |
| … | | | |

**제외한 키** — 이 게임(PC, 키보드+마우스)에 절대 표시되지 않으므로 일본어 그대로 둔다.

- VR: Mixed Reality(11554–11595), Oculus Touch(11604–11641)
- 모바일/콘솔: Android(11466–11469), Steam 컨트롤러(11657–11662), Touch

> 굳이 안 보이는 키까지 번역하면 오역 위험만 늘고 이득이 없다. "보이는 것만 고친다"는 원칙.

## 3. 반전: 게임은 "짧은 이름(Short)"을 쓴다

`スペースバー`(SpaceBar)를 `스페이스바`로 바꿨는데도 복호화 프롬프트는 여전히 **`スペース`** 로 떴다. `スペースバー`(긴 이름)와 `スペース`(짧은 이름)는 **다른 문자열**이다.

언리얼의 `FKey`는 **긴 표시명(DisplayName)과 짧은 표시명(ShortDisplayName)** 을 따로 가진다. 게임의 키 프롬프트 위젯은 **짧은 이름이 있으면 짧은 이름**을 쓴다.

`スペース`가 어느 항목인지 확정하기 위해, locres 엔트리 영역에서 **(키 문자열 → 인덱스)** 를 brute-scan 했다. 엔트리 한 칸은 `keyHash(4) + FString(키 문자열) + srcHash(4) + int32 index` 구조다([02 문서](02-reverse-engineering.md)의 키 블록 참조).

```
[4908] 'スペース'  ← key = "SpaceBarShort"   (SpaceBar의 짧은 표시명)
[11656] 'スペースバー' ← key = "SpaceBar"      (긴 표시명, 1차에서 번역함)
[11552] '左 Shift'  ← key = "LeftShift"
```

즉 게임은 Space에 대해 **`SpaceBarShort`[4908]** 를 표시하고 있었고, 이 인덱스는 11461–11669 블록 **밖**이라 1차에서 놓쳤다. 반면 Shift/Ctrl/Alt는 짧은 이름이 없어 긴 이름(`左 Shift`)을 쓰므로 1차 번역으로 이미 해결돼 있었다.

### 추가로 번역한 짧은 키

| 키 | 인덱스 | 일본어 | 한국어 |
|---|---|---|---|
| `SpaceBarShort` | 4908 | スペース | 스페이스 |
| `Up` | 4485 | 上 | 위 |
| `Down` | 4486 | 下 | 아래 |
| `Left` | 5953 | 左 | 왼쪽 |
| `Right` | 5961 | 右 | 오른쪽 |
| `Up`(여백 포함) | 7538 | `上␣␣␣` | `위␣␣␣` |
| `Home` | 11542 | ホーム | 홈 |

### dedup 주의

locres는 **동일 문자열을 하나로 합쳐(dedup)** 여러 키가 같은 인덱스를 참조한다. 예를 들어 인덱스 5953 `左`는 키 `Left`(방향키)뿐 아니라 `CameraMenuTitle_Left`, `LeftText`, `TextJustify::Left` 등 에디터 문자열도 함께 가리킨다. 이 인덱스를 `왼쪽`으로 바꾸면 그 모든 키가 함께 바뀌지만, **에디터 문자열은 런타임에 표시되지 않으므로 인게임에는 무해**하다.

최종적으로 엔진 키 이름 **115개**(긴 이름 109 + 짧은/방향키 등)를 한국어화했다. 매핑은 다음 파일에 있다.

```
data/engine_keys.json   (일본어 → 한국어, 115개; 빌드 시 Engine.locres 에 적용)
data/engine_keys.tsv    (인덱스·일본어·한국어, 사람 검토용)
```

## 4. 원작 버그: 리터럴 `\r`

복호화 화면과 별개로, 한 통신 메시지에서 본문 중간에 **`\r` 두 글자가 그대로** 보였다.

> 백토끼 신호로 인한 통신 장애가 있다고는 해도 **\r**벌써 12 시간 넘게 …

원인을 추적하니 **원본 일본어 자체의 버그**였다. 해당 문자열의 일본어 원문에는 실제 캐리지 리턴(CR, `0x0D`)이 아니라 **리터럴 백슬래시 + r (2글자)** 가 들어 있었다.

```
JA(원본): '白ウサギ信号による通信障害があるとはいえ、\rもう12 時間以上も動きがない。'
                                                  ^^  ← 실제 CR이 아니라 글자 '\','r'
```

**줄바꿈 처리는 위젯마다 다르다.** 처음에는 일기/문서 위젯에서 줄바꿈으로 동작하는 게임 고유 토큰 `{/n}`으로 교정했다. 그런데 **통신(메일) 위젯에서는 `{/n}`마저 글자 그대로** 출력됐다(메시지 128에서 발견). 이 위젯은 토큰을 무시하고 텍스트를 **자동 줄바꿈**한다.

| 표현 | 일기/문서 위젯 | 통신(메일) 위젯 |
|---|---|---|
| 리터럴 `\r` (백슬래시+r) | 글자로 출력 ❌ | 글자로 출력 ❌ |
| `{/n}` 토큰 | 줄바꿈 ✅ | **글자로 출력 ❌** |
| 공백 + 자동 줄바꿈 | 같은 줄(또는 자연 감김) | 자연 감김 ✅ |

그래서 **최종 교정은 "공백 교체"**다. 버그가 있던 문자열(원문에 리터럴 `\r`이 있는 항목)의 줄바꿈을 공백으로 바꾸면 어느 위젯에서도 토큰이 노출되지 않고 자동 줄바꿈으로 자연스럽게 감긴다. 원본부터 `{/n}`을 쓰던 **정상 문자열(일기/문서)은 건드리지 않도록**, ja 원문에 리터럴 `\r`이 있는 항목만 정밀 타겟했다.

해당 인덱스: `192, 222, 389, 504, 522, 528, 537, 611, 693, 933, 1115, 1140, 1263` (이 중 `{/n}`을 넣었던 11건을 공백으로 교체; 222의 정상 `\r\n` 단락 구분은 보존).

이 교정은 `data/translations.json`의 한국어 값에 그대로 반영돼 있어, 빌드하면 자동 적용된다.

> 교훈: ① 원작에 소스 버그가 있으면 충실한 번역은 그 버그까지 복제한다. ② **한 위젯에서 통한 줄바꿈 방식이 다른 위젯에서도 통한다고 가정하지 말 것** — 가장 안전한 건 공백 + 엔진 자동 줄바꿈이다.
