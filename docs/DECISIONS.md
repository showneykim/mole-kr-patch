# 의사결정 로그 (Decision Log)

> **요약**: 이 패치를 만들며 내린 핵심 기술 결정들을 ADR(Architecture Decision Record) 형식으로 기록한다. 각 항목은 *맥락 / 결정 / 근거 / 대안 / 결과* 순서다.

---

## D1. 일본어(ja) 로케일 슬롯 하이재킹

- **맥락**: 게임은 `ko` 로케일이 없고, 한국어를 정식 추가하려면 언어 목록 UI·컬처 매핑·폰트 매핑까지 수정해야 한다.
- **결정**: 사용되지 않는 **일본어 슬롯을 한국어로 덮어쓴다.** 사용자는 게임 언어를 `日本語`로 설정한다.
- **근거**: locres 텍스트와 폰트, **두 종류 파일만** 교체하면 된다. 모든 언어가 동일한 키 구조를 공유하므로 ja 슬롯만 바꿔도 다른 언어는 멀쩡하다.
- **대안**: 신규 `ko` 로케일 추가 → IoStore 안의 위젯/설정/컴포지트 폰트까지 손대야 해 복잡·취약.
- **결과**: 채택. 가장 미니멀하고 게임 업데이트에 견고.

## D2. 레거시 pak `_P` 패치 오버라이드 (IoStore 직접 수정 회피)

- **맥락**: 게임은 UE5 IoStore(`.ucas`/`.utoc`) + 레거시 `Mole-Windows.pak`을 함께 쓴다. IoStore 수정은 어렵다.
- **결정**: 텍스트·폰트가 **레거시 pak 안에 loose 파일로** 들어있음을 확인하고, 같은 경로를 담은 **`_P` 패치 pak**으로 오버라이드한다.
- **근거**: `Game.locres`, `Engine.locres`, `.ufont`(폰트 원본 TTF)가 전부 레거시 pak에 있었다(IoStore에는 uasset만 있음). `_P` 접미사 pak은 우선순위가 높아 동일 경로를 덮어쓴다.
- **대안**: `retoc`로 IoStore를 풀고 다시 패킹 → 훨씬 복잡. (검증용으로만 사용)
- **결과**: 채택. `repak`만으로 추출/패킹 가능. 빌드가 단순해짐.

## D3. 직접 작성한 locres v3 파서 — 문자열 테이블만 재작성

- **맥락**: `Game.locres`는 UE LocRes **v3(Optimized_CityHash64_UTF16)** 포맷. 표준 도구(UnrealLocres 등)는 Windows/.NET 의존.
- **결정**: Python으로 직접 파서(`tools/locres.py`)를 작성하되, **헤더 + 엔트리(네임스페이스/키 테이블)는 불투명 바이트로 그대로 보존**하고 **뒤쪽 문자열 테이블만 다시 쓴다.**
- **근거**: 번역 텍스트는 전부 문자열 테이블에 있고 엔트리는 인덱스로만 참조한다. 엔트리 포맷이 비표준이라 완전 파싱이 까다로웠지만, 이 방식이면 **원본을 바이트 단위로 무손실 라운드트립**할 수 있어 안전하다.
- **대안**: 엔트리까지 완전 파싱 → 실패 위험·불필요. (단, en↔ja 정렬을 위해 GUID 키 블록은 **스캔**으로 추출했다 — [02](02-reverse-engineering.md) 참조)
- **결과**: 채택. 1300개(Game) / 34,552개(Engine) 모두 라운드트립 무손실 확인.

## D4. 폰트 6종 전부 교체

- **맥락**: 한글 글리프가 없어 텍스트만 바꾸면 `□`(두부)로 깨진다. 게임은 화면 요소마다 **서로 다른 폰트**를 쓴다.
- **결정**: 커스텀 폰트 **6종 모두**(`DotGothic16-Regular`, `ZLabsBitmap_12px_CN/HC/HC_FALLBACK`, `MonocraftUpdated`, `small_adventure_inverted_font`)를 한글 폰트로 교체.
- **근거**: 시행착오로 확인 — 일부만 바꾸면 로딩 경고문은 한글이 나와도 **타이틀 메뉴(다른 폰트)**는 여전히 `□`. 빌드 표시("빌드")는 영문 전용 `MonocraftUpdated`였다.
- **대안**: 게임이 쓰는 폰트만 골라 교체 → 어떤 UI가 어떤 폰트를 쓰는지 일일이 추적해야 해 비효율·누락 위험.
- **결과**: 채택. 모든 UI에서 한글 렌더 확인. (`.ufont`은 크기와 무관하게 통째로 로드됨 — [03](03-fonts.md))

## D5. 한글 폰트: NeoDunggeunmo → Galmuri11

- **맥락**: 첫 시도로 NeoDunggeunmo(둥근모, 651KB)를 원본과 같은 크기로 패딩해 넣었더니 **로딩 경고문(한자 포함)까지 깨졌다.**
- **결정**: **Galmuri11**(한글+영문+가나+한자 2350+, 도트체)로 교체.
- **근거**: NeoDunggeunmo는 **한자가 없어** `警告` 같은 글자가 깨졌다. Galmuri11은 한자까지 커버해 **전환기에 미번역 일본어(가나·한자)가 남아도 깨지지 않는다.** 레트로 픽셀 분위기도 부합.
- **대안**: Galmuri11Bitmap(임베디드 비트맵, 특정 px 전용) → 렌더 크기 의존 위험. NeoDunggeunmo → 한자 미커버.
- **결과**: 채택. Galmuri는 SIL OFL 1.1로 재배포 가능.

## D6. 번역: ja 우선 + en 참고, 멀티에이전트 3렌즈 교차검증

- **맥락**: 1300개 문자열(서사 비중 큼). 품질·일관성이 중요.
- **결정**: 우리가 채우는 슬롯이 ja이므로 **일본어 의미 우선, 영어는 보조 참고.** 카테고리(UI/시스템/아이템/대사/일기/문서 등)별로 묶어 **번역 → 3렌즈(정확성/용어·문체/태그·자연스러움) 병렬 교차검증 → 교정** 워크플로로 진행.
- **근거**: en은 1795개로 ja(1300)와 개수·정렬이 달라 인덱스로 못 맞춘다. **키 GUID로 en↔ja를 조인**해 영어 원문을 정확히 가져왔다. 공식 스팀 한국어 용어(두더지/굴착기/항해사/승조원…)를 글로서리로 고정.
- **대안**: en→ko 단독 번역 → 슬롯이 ja라 원문 대조가 어긋남. 단일 패스 → 일관성·품질 검증 부족.
- **결과**: 채택. 고유명사 불일치 0건, 태그 보존 위반 0건(프로그램 검증).

## D7. 배포 데이터는 index가 아니라 ja 원문으로 키잉

- **맥락**: 번역은 인덱스 단위로 했지만, 인덱스는 게임이 문자열을 추가/삭제하면 어긋난다.
- **결정**: 배포용 `data/translations.json`을 **`일본어 원문 → 한국어` 매핑**으로 저장한다. 빌드 시 사용자 게임의 ja 문자열을 보고 매칭한다.
- **근거**: ja 1300개가 **전부 고유**하고(중복 0) ja→ko 충돌이 **0건**이라 안전한 1:1 매핑이 된다. 게임 업데이트로 인덱스가 밀려도, 원문이 그대로면 계속 적용된다. 사람이 읽고 검토하기도 쉽다.
- **대안**: index→ko(인덱스 변동에 취약), GUID→ko(불투명·개발자가 키 재생성 시 전멸).
- **결과**: 채택. 검토용 `data/translations.tsv`(index, guid, en, ja, ko)도 함께 제공.

## D8. 원작 버그(리터럴 `\r`)는 위젯에 맞게 교정 — 줄바꿈 처리는 위젯마다 다르다

- **맥락**: 일부 원본 일본어 문자열에 **실제 CR(0x0D)이 아닌 리터럴 백슬래시+r(2글자)**가 들어 있다. 게임은 이를 줄바꿈으로 렌더하지 못하고 글자 그대로 `\r`을 보여준다 — **원작의 로컬라이즈 버그.**
- **결정(1차)**: 리터럴 `\r`을 게임의 줄바꿈 토큰 `{/n}`으로 교체. (일기/문서 위젯에서 `{/n}`이 줄바꿈으로 동작함을 원본에서 확인했기 때문.)
- **반례**: 그런데 **통신(메일) 위젯은 `{/n}`을 해석하지 않고 글자 그대로 출력**했다(메시지 128에서 발견). 즉 **줄바꿈 처리는 위젯마다 다르다** — 이 위젯은 토큰을 무시하고 텍스트를 **자동 줄바꿈**한다.
- **결정(최종)**: 버그가 있던 문자열(원문에 리터럴 `\r`)의 줄바꿈을 **공백으로 교체**한다. 자동 줄바꿈 위젯에서는 공백이 자연스럽게 감기고, 어느 위젯에서도 토큰이 글자로 노출되지 않는다. 일기/문서 위젯에서 **정상 동작하는 `{/n}`(원본부터 쓰던 것)은 건드리지 않는다**(데이터에서 ja 원문에 리터럴 `\r`이 있는 항목만 정밀 타겟).
- **근거**: 원작자의 의도는 "부드러운 줄바꿈"이었고, 모든 위젯에서 안전하게 동작하는 표현은 **공백 + 자동 줄바꿈**이다.
- **결과**: 채택. 교훈 — *한 위젯에서 통했다고 모든 위젯에서 통한다고 가정하지 말 것.*

## D9. 배포: 프리빌트 pak + 빌드 스크립트 둘 다, 게임 에셋 미포함

- **맥락**: 오픈소스로 공개하되 법적으로 깔끔하게.
- **결정**: **프리빌트 `Mole-Windows_P.pak`**(초간단 설치)과 **빌드 스크립트**(`tools/build.py`, 재현성) 둘 다 제공. 저장소에는 **게임 원본 에셋을 포함하지 않는다.** 프리빌트 pak은 우리 번역문 + 자유 폰트로만 구성.
- **근거**: 일반 사용자는 파일 하나만 받아 드롭하면 되고, 개발자는 자기 게임에서 직접 빌드해 출처를 검증할 수 있다.
- **대안**: 프리빌트만(재현성·투명성 부족) / 빌드 스크립트만(일반 사용자 진입장벽).
- **결과**: 채택. 번역문은 원작의 2차적 저작물이므로 게임 정식 소유자 전용·비상업으로 제공([DISCLAIMER](../DISCLAIMER.md)).
