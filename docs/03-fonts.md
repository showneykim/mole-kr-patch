# 03. 폰트 — 이 프로젝트 최대의 난관

> 요약: M.O.L.E은 한글 글리프가 없는 **픽셀/비트맵 폰트**를 쓴다. 텍스트만 한국어로 바꾸면 글자가 `□`/`◆◆`로 깨진다. 게임이 **UI 요소마다 다른 폰트**를 쓰기 때문에, 결국 커스텀 폰트 `.ufont` **6종을 전부** 한글 도트 폰트(Galmuri11)로 교체해야 했다. 그 과정이 이 프로젝트에서 가장 많은 시행착오를 낳았다.

관련 문서: 텍스트 교체 자체는 [02-reverse-engineering.md](02-reverse-engineering.md), 의사결정 요약은 [DECISIONS.md](DECISIONS.md) 참조.

---

## 1. 문제: 한글 글리프가 없다

번역(locres)만 교체하고 게임을 켜자, 한국어로 바꾼 메뉴가 전부 `◆◆`(빈 네모/마름모)로 표시됐다. 텍스트 데이터는 정상적으로 한국어로 바뀌었지만, **게임 폰트에 한글(한글 자모 조합 11,172자) 글리프가 존재하지 않아** 렌더링 단계에서 깨진 것이다.

이 게임은 레트로 터미널 분위기의 **픽셀(도트) 폰트**를 사용한다. 한글은 조합형 문자라 폰트가 한글 영역을 포함하지 않으면 어떤 글자도 그릴 수 없다.

## 2. 폰트 자산의 정체

폰트는 `Mole/Content/Drift/Art/Fonts/` 아래 `.ufont` 6개로 존재하며, **IoStore(.ucas)가 아니라 레거시 `Mole-Windows.pak`에 loose 파일로** 들어 있다(이 점이 결정적으로 작업을 쉽게 만들었다).

| `.ufont` | 용도(추정) | 원본 크기 |
|---|---|---|
| `DotGothic16-Regular` | 일본어(가나/한자) | 약 2.03 MB |
| `ZLabsBitmap_12px_CN` | 중국어 전용 비트맵 | 약 3.33 MB |
| `ZLabsBitmap_12px_HC` | 본문 비트맵(주력) | 약 2.68 MB |
| `ZLabsBitmap_12px_HC_FALLBACK` | 폴백 비트맵 | 약 3.33 MB |
| `MonocraftUpdated` | 영문/숫자(모노스페이스) | 약 0.21 MB |
| `small_adventure_inverted_font` | 장식용 | 약 0.018 MB |

`.ufont`의 첫 4바이트는 `00 01 00 00` — 즉 **UE 컨테이너 헤더 없이 원시 TrueType(sfnt) 폰트**다. FontFace `.uasset`은 IoStore에 있지만, 폰트 본체(TTF 바이트)는 이 `.ufont`이고 런타임에 파일째 로드된다. 따라서 **`.ufont`을 한글 TTF로 덮으면** 해당 폰트가 한글을 그릴 수 있게 된다.

```
$ xxd DotGothic16-Regular.ufont | head -1
00000000: 0001 0000 0010 0100 0004 0000 4241 5345   ..........BASE
#         ^^^^^^^^^ = sfnt 1.0 (TrueType)        ^^^^ = 'BASE' 테이블
```

## 3. 시행착오 — 다섯 단계

폰트 문제는 한 번에 풀리지 않았다. 실제로 거친 순서:

### (1) 텍스트만 교체 → `◆◆`
한국어 locres만 적용. 메뉴의 한글이 전부 빈 네모로. → **폰트에 한글 글리프가 없음**을 확인. 폰트 교체가 필요하다는 결론.

### (2) DotGothic16 하나만 Galmuri로 교체 → 일부만 반응
일본어 폰트인 `DotGothic16`만 한글 폰트로 바꿨다. 결과:
- **로딩 화면의 일본어 경고문은 폰트를 따라 바뀜** → `.ufont` 교체가 실제로 먹힌다는 증거.
- **하지만 타이틀 메뉴의 한글은 여전히 `◆◆`로 깨짐.**

→ 핵심 발견: **게임은 화면/요소마다 다른 폰트를 쓴다.** 로딩 경고문과 타이틀 메뉴가 서로 다른 폰트를 참조한다. 메뉴가 쓰는 폰트(`ZLabsBitmap` 계열 또는 `MonocraftUpdated`)는 아직 한글이 없었던 것.

증거 화면: 우하단 빌드 표기가 `◆◆ 1.1.5`로, 바로 옆 숫자 `1.1.5`는 멀쩡 → 그 텍스트는 **영문 전용 폰트**를 써서 한글만 깨진 상황이었다.

### (3) "크기 검증" 가설 — 오류로 판명
한때 "UE가 `.ufont`의 저장된 크기만큼만 읽어서, 원본보다 큰 폰트는 잘려 깨진다"고 의심했다. **틀렸다.** UE는 `.ufont`을 **파일째(whole-file) 로드**하며 크기와 무관하다.

> 결정적 증거: 원본 2 MB 슬롯(`DotGothic16`)을 **5.3 MB짜리 Galmuri11**로 덮었는데도 일본어가 멀쩡히 렌더됨. 잘렸다면 TTF가 깨져 일본어도 안 나왔어야 한다. → 크기 무관, 통째 로드 확정.

이 깨달음 덕분에 이후로는 폰트 크기를 맞출 필요 없이 자유롭게 교체했다.

### (4) NeoDunggeunmo 시도 → 한자가 없어 실패
레트로 도트 미감이 좋은 **NeoDunggeunmo(neodgm, 651 KB)**를 (당시엔 크기 가설 때문에) 원본 크기로 0 패딩해 넣었다. 결과: 한글은 나오는데, **`警告` 같은 한자가 없어서** 미번역 일본어(로딩 경고문 등)까지 깨졌다.

→ 교훈: 전환기에는 **미번역 일본어가 남아 있으므로**, 한글만 있는 폰트로는 부족하다. **한글 + 가나 + 한자**를 모두 커버하는 폰트가 필요하다.

### (5) 6종 전부 Galmuri11로 교체 → 성공
커스텀 폰트 **6종 전부**를 `Galmuri11.ttf`로 교체했다. 어떤 UI가 어떤 폰트를 참조하든 한글이 그려지게 된다. Galmuri11은 **한글 + 영문(Latin) + 가나 + 한자(2350+)**를 모두 포함하므로, 한국어는 물론 아직 번역 안 된 일본어도 깨지지 않는다.

결과: 타이틀의 `플레이/설정/종료`, 우하단 `빌드 1.1.5`, 본문·아이템·터미널까지 전부 한글 도트체로 정상 렌더. 폰트 난관 돌파.

## 4. 교훈 정리

- **UI 요소마다 폰트가 다를 수 있다.** 한 폰트만 바꾸면 일부 화면만 한글이 된다. **관련 커스텀 폰트를 모두 교체**해야 한다.
- **`.ufont`은 크기 무관, 파일째 로드.** 교체 폰트 크기를 원본에 맞출 필요가 없다.
- **전환기 안전성을 위해 폭넓은 커버리지 폰트를 쓴다.** 한글 전용 폰트는 미번역 일본어(특히 한자)를 깨뜨린다. 한글+가나+한자 폰트가 안전하다.

## 5. 폰트 선택: Galmuri11

- **이유**: 레트로 도트(픽셀) 미감이 게임의 터미널 분위기와 일치하고, 한글·영문·가나·한자까지 커버리지가 넓다.
- **라이선스**: SIL Open Font License 1.1. 전문은 [`fonts/Galmuri-OFL.txt`](../fonts/Galmuri-OFL.txt). © Yang Hee-jun(quiple), https://galmuri.quiple.dev
- 본 패치는 Galmuri11을 `.ufont` 컨테이너로 재배치하여 사용한다(OFL의 임베딩 허용 범위 내).

## 6. 빌드에서의 처리

[`tools/build.py`](../tools/build.py)는 한 개의 한글 TTF를 6개 `.ufont` 경로에 모두 복사한다.

```python
FONTS = [
    "DotGothic16-Regular", "ZLabsBitmap_12px_CN", "ZLabsBitmap_12px_HC",
    "ZLabsBitmap_12px_HC_FALLBACK", "MonocraftUpdated", "small_adventure_inverted_font",
]
# 각 폰트를 동일한 한글 폰트로 교체
for f in FONTS:
    shutil.copy(args.font, os.path.join(stage, FONT_DIR, f + ".ufont"))
```

다른 한글 폰트를 쓰고 싶다면 `--font /경로/한글폰트.ttf`로 교체 가능하다(단, 한글+가나+한자 커버리지를 권장).
